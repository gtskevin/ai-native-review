# AI-Native 设计原则（全局规则）

## 为什么需要这个规则

AI 构建工具时默认走规则路线（训练偏差），会把"自己"排除在执行层之外：写好工作流，但用 if/else/regex 替代自己的推理能力。结果是"聪明人指挥傻子干活"。这个规则强制 AI 在核心工作上使用自己的推理能力。

## 核心定义

AI-Native ≠ 全部用 AI。AI-Native = **AI 做核心判断，代码做精确执行，人做关键确认，三者协作**。

完整审查需要区分两个互补维度：

| 维度 | 核心问题 | 健康模式 |
|------|----------|----------|
| **AI-native 智能内核** | 需要理解的判断是否真的由模型完成？ | LLM 判断 + 证据 + 置信度 |
| **AI-native 交互** | 用户能否表达目标，让系统在适当监督下规划和执行？ | 目标导向输入 + 动态编排 + 过程可见 + 关键确认 |

两个维度必须分别判断。不要把它们平均成一个分数：有聊天框不代表核心判断是 AI 驱动；模型能力很强也不代表用户交互足够自然。

---

## 1. 决策框架：每个子任务怎么分配

对每个子任务问三个问题：

1. **需要"理解和判断"吗？** → AI（LLM / embedding / 分类模型）
2. **需要"一致性和可预测性"吗？** → 确定性代码
3. **需要"权威数据"吗？** → RAG / 工具调用

然后：
4. **LLM 输出是否可检验？** → 设计结构化输出 + 证据引用
5. **判断出错的代价多大？** → 高代价加人在回路
6. **总成本/延迟是多少？** → 优化计算预算

---

## 2. 能力分工

**AI 做（需要判断力的）：**
- 内容评估（"这个问题是否有引导性"）
- 推理决策（"应该怎么做"、"哪个方案更好"）
- 语义理解（"用户真正想要什么"）
- 创意生成（"写出"、"设计"、"建议"）

**代码做（需要精确性的）：**
- 数学计算、金额统计、CRUD 操作
- 格式转换、类型校验、输出结构化
- 流程安全（输入验证、注入防护、重试逻辑）

**人做（需要权威的）：**
- 高风险决策的最终确认
- AI 输出的审阅和否决
- 涉及安全、金额、发布等不可逆操作

---

## 3. 架构模式：从简到繁

| 模式 | 适用场景 | 典型例子 |
|------|---------|---------|
| **直接调用** | 单步生成/评估 | 文本评分、内容检查、问卷审阅 |
| **LLM-as-a-Judge** | 质量评判 | 对比评估、打分、审稿 |
| **链式调用** | 上一步输出是下一步输入 | 摘要→翻译→格式化 |
| **RAG** | 需要知识库支撑 | 基于文档的问答 |
| **Reflection** | 需要自检 | LLM 检查自己的输出 |
| **Agent** | 需要自主选择策略 | 复杂分析、多步骤任务 |
| **多 Agent** | 需要协作与互相验证 | 一个生成一个审查 |

**原则：** 大多数 Skill 和工具用"直接调用"或"LLM-as-a-Judge"就够了。

---

## 4. 可靠性设计

LLM 输出是概率性的，必须设计保护层：

1. **结构化输出** — JSON/schema，不要纯自然语言
2. **证据引用** — 每个判断附带指向具体原文的证据
3. **置信度** — 标注判断的把握程度
4. **验证/自检** — 对关键输出使用 Reflection 模式二次验证
5. **空结果 = 失败** — LLM 返回空时不标记为成功
6. **fallback 解析** — JSON 解析失败时用兜底方案，不要 crash

---

## 5. 成本与延迟意识

- **AI ≠ 只有 LLM** — embedding 做语义匹配（比规则好，比 LLM 便宜）、分类模型做意图识别（比规则好，比 LLM 快）
- **不所有任务都用最贵模型** — routine 用便宜/快速模型，复杂推理用强模型
- **缓存 AI 输出** — 相同输入 + 相同模型 = 缓存而不是重复调用
- **流式输出** — 响应超过 2 秒时必须 streaming + 骨架屏

## 6. 交互设计：目标 → 规划 → 执行 → 确认

AI-Native 的交互模式根本不同：

| 传统 | AI-Native |
|------|-----------|
| 用户理解流程并逐步点击 | 用户描述目标，AI 规划并执行 |
| 一次性输出 | 支持追问、纠正和迭代 |
| 用户只看结果 | 用户看到进度、证据和已执行动作 |
| 所有动作自动执行或所有动作都阻塞 | 低风险操作自动化，高风险操作请求确认 |

参考流程：

```text
用户用自然语言表达目标
        ↓
AI 澄清缺失约束
        ↓
AI 提出计划，或在清晰边界内形成计划
        ↓
AI 调用工具并报告进展
        ↓
低风险、可逆操作 → 直接执行
高风险或不可逆操作 → 请求确认
        ↓
用户审阅、纠正、覆盖决定，或继续对话
```

逐项检查：

| 领域 | 检查问题 | 健康证据 |
|------|----------|----------|
| **意图表达** | 用户能否直接描述想要的结果，而不必理解步骤、API 或菜单？ | 自然语言目标、上下文默认值 |
| **动态编排** | 模型能否根据当前状态选择下一步或合适工具？ | 动态规划和工具选择 |
| **过程可见** | 用户能否知道系统正在做什么、已经改变了什么？ | 进度、证据、操作日志、明确结果 |
| **可纠正** | 用户能否细化、中断、撤销、重试或接管？ | 追问、编辑、取消、回滚 |
| **高风险确认** | 对有后果的操作是否设置适当门槛？ | 确认、有限权限、人工接管 |
| **界面匹配** | 是否在自然语言能减少摩擦时使用对话，同时保留有助于精确操作的结构化 UI？ | 对话 + 预览、表单、表格或确认控件 |

**边界：** 不要为了 AI-native 强行加入聊天框。自然语言适合表达模糊、多步骤目标；表单、表格、直接操作和审批控件仍然适合精确输入、比较、审阅和确认。

**对 Skill 的启示：** 优先允许用户用自然语言描述任务（prompt-as-input），减少配置负担。用户应该能看到 AI 为什么这么判断，并能否决或调整。

## 7. 反模式检测

写代码时遇到以下模式，标记为 AI-decorated 并建议重构：

| 反模式 | 你正在写的代码 | 应该改成 |
|--------|-------------|---------|
| **AI 编排 + 规则执行** | AI 设计流程，核心判断用 regex/if-else | AI 做核心判断 |
| 正则代替判断 | `regex.test(content)` 检查质量 | LLM 评估质量 |
| 模板代替生成 | `template.replace('{x}', value)` | LLM 根据上下文生成 |
| if-else 做业务决策 | `if (score > 0.8) good else bad` | LLM 解释为什么好/不好 |
| 关键词匹配 | `text.includes('关键词')` | embedding 语义匹配 或 LLM 理解 |
| 点击迷宫 | 用户手动执行本可由 agent 规划的目标导向步骤 | 用户表达目标，AI 编排执行 |
| 聊天框装饰 | 有对话入口，但核心决策和下一步仍然固定 | 模型根据状态选择动作 |
| 无边界自主性 | 未确认就执行有后果的操作 | 设置确认、权限范围和回滚 |
| 隐藏执行过程 | 用户不知道系统做了什么 | 显示进度、证据和动作结果 |

**注意：** Prompt 本身也是一种"软规则"——你在 prompt 中写的审阅标准本质上也是规则，只是比硬编码灵活得多、会随模型升级而提升。目标不是消灭规则，是用"活的软规则"替代"死的硬规则"。

## 8. 设计清单（每个新工具/Skill 开始前）

```
第一步：任务分解
  - 列出所有子任务
  - 标记每个任务的性质（判断/搬运/格式化/认证/存储）

第二步：能力分配
  - 判断性任务 → AI（选择合适的 AI 工具：LLM/embedding/分类模型）
  - 确定性任务 → 代码
  - 高风险判断 → AI + 人在回路

第三步：可靠性设计
  - AI 输出需要结构化（JSON/schema）
  - 每个判断附带证据和置信度
  - 高 stakes 场景加 Reflection 自检

第四步：成本优化
  - 评估 AI 调用次数和 token 消耗
  - 考虑用 embedding/小模型替代部分 LLM 调用
  - 设计缓存策略

第五步：交互设计
  - 用户能否直接表达目标，而不必理解内部流程？
  - AI 能否根据当前状态动态规划并选择工具？
  - 用户如何看到 AI 的进度、动作结果、判断过程和证据？
  - 用户如何否决、调整、中断、撤销或接管？
  - 哪些低风险动作可自动执行？哪些高风险动作必须确认？
  - 哪些场景应该保留表单、表格、预览或审批控件？

第六步：验证 AI-native
  - 对核心功能做删除测试（删掉 AI 后还剩什么？）
  - 模型升级测试（更好的模型会让性能提升吗？）
  - 交互测试（用户能否通过“目标 → 规划 → 执行 → 确认”减少不必要操作？）
  - 边界测试（是否为了 AI-native 强行加入聊天框，或让 agent 获得无边界自主权？）
  - 分别给出“智能内核”和“交互”结论，不要平均成一个总分
```
