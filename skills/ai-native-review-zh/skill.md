# AI-Native 设计审查

## 触发时机

- 开始设计新 Skill、工具、Web App 或功能前
- 用户要求审查已有工具是否 AI-native
- 用户说"帮我检查这个是不是 AI-native"、"审查一下设计方案"

## 审查流程

### 第零步：快速筛查

三个问题任一为"是"则跳过完整审查：

1. **核心功能是纯确定性操作？**（文件管理、格式转换、CRUD）→ 不需要审查
2. **处于合规/安全/性能关键路径？**（合规检查、实时 API）→ 规则优先，不适用
3. **复杂度低？**（< 3 个子任务，无语义需求）→ 不值得审查

通过筛查 → 进入完整审查。

### 第一步：理解任务

- 核心任务是什么？（一句话）
- 目标用户是谁？
- 预期输出是什么？

### 第二步：任务分解

| 性质 | 标记 | 应该用 |
|------|------|--------|
| 需要理解和判断 | 🧠 | AI（LLM/embedding/分类） |
| 需要精确执行 | ⚙️ | 确定性代码 |
| 需要权威数据 | 📚 | RAG/工具调用 |
| 高风险不可逆 | 👤 | 人在回路 |

### 第三步：健康检查（决策树）

对每个 🧠 子任务，按顺序通过以下决策树：

```
删除测试：删掉 AI 后还能完成吗？
├── 能 → ❌ AI-decorated，跳到下一个子任务
└── 不能 → 继续 ↓

模型升级测试：更强模型会让结果更好吗？
├── 不会 → ❌ 核心不是 AI 做的
└── 会 → 继续 ↓

上下文充分性测试：AI 获得了做出判断所需的全部信息吗？
├── 输入不完整 → ⚠️ AI-native 但上下文不足，标注缺少什么
├── prompt 质量差（模糊的"请评估"）→ ⚠️ 需改进 prompt
├── 管线有信息丢失 → ⚠️ 修复管线
└── 充分 → 继续 ↓

软规则测试：prompt 是否比硬编码规则更灵活？
├── 硬规则换壳 → ❌
└── 活的软规则 → ✅，但额外检查：
    ├── 用户输入影响 prompt 构造？→ 标注"需输入隔离"
    ├── 需要可复现？→ 标注"需确定性回退"
    └── 模型升级可能改变行为？→ 标注"需回归测试"
```

**混合结果：** 核心子任务通过 → 整体 ✅；核心未通过 → 整体 ❌；无法判定 → 标注"需人工判定"。

### 第四步：反模式扫描 + 领域豁免

检查以下反模式：

| 反模式 | 检测方法 |
|--------|---------|
| AI 编排 + 规则执行 | AI 设计流程，核心判断用 regex/if-else |
| 正则代替判断 | regex 检查内容质量/语义 |
| 模板代替生成 | 字符串拼接而非 LLM 生成 |
| if-else 做业务决策 | 阈值判断好坏而非 AI 解释 |
| 关键词匹配代替理解 | includes/match 而非语义匹配 |
| 纯自然语言输出 | 无结构化、无证据、无置信度 |

**领域豁免：** 合规/审计、安全扫描、实时/低延迟、数据脱敏、金额计算——这些场景中规则是正确选择，标注为"合理使用规则"。

### 第五步：开放式判断

Step 1-4 未覆盖的问题：

1. AI 使用方式是否与用户场景匹配？有没有"不该用 AI 的地方用了"或"该用没用"？
2. 有数据飞轮吗？使用越多 AI 是否越好？
3. 最可能的失败模式是什么？（幻觉、偏见、延迟、成本）用户能感知吗？
4. 有没有前面步骤遗漏的问题？

将发现加入报告"补充发现" section。没有则写"无补充发现"。

### 第六步：输出报告 + 自检

```
## AI-Native 设计审查报告

**工具名称：** xxx  **核心任务：** xxx

### 子任务分解
| 子任务 | 性质 | 实现方式 | AI-native? | 上下文充分? |

### 发现的问题
每个发现包含：问题简述 | 发现来源（哪个 Step） | 证据 | 严重度 | 建议修改

### 补充发现
（Step 5 的额外发现，或"无补充发现"）

### 行动建议
| 优先级 | 问题 | 改动范围 | 风险 |
|--------|------|---------|------|
| P0 立即改 | | 核心重写 | 需回归测试 |
| P1 近期改 | | 增加机制 | 不影响现有 |
| P2 可选 | | 边缘改进 | 低 |

### AI-Native 值不值？
| 🧠 子任务 | 规则替代代价 | AI 运行成本 | 维护负担 | 值不值？ |
（"规则替代代价=轻微"且"成本≠0" → ⚠️ 需论证必要性）

### 架构 & 可靠性
- 推荐模式：[直接调用 / LLM-as-a-Judge / 链式 / RAG / Reflection / Agent]
- 可靠性：结构化输出 | 证据引用 | 置信度 | 人在回路

### 漂移风险
最容易被后续"加规则"退化的子任务 → 标记敏感点
```

**自检（5+ 个 🧠 子任务时执行）：**
- Step 2 是否覆盖所有子任务？Step 3 是否覆盖所有 🧠？
- 校准对照：regex 查质量 → 应识别；LLM 做数学 → 应标记错配；模板拼 LLM → 应识别
- 总体信心：高/中/低

报告不是终点——用户可追问任何发现的置信度、是否误报、或重新排序优先级。

### 第七步：如果发现 AI-decorated

给出重设计方案：
1. 哪些子任务应该 AI 做但现在用规则做了
2. 具体 prompt 设计建议（输入输出格式）
3. 哪些规则应保留为护栏
4. 分阶段迁移路径（Phase 1 最小改造 → Phase 2 核心替换 → Phase 3 可靠性增强），每阶段有验证点和回滚方案
5. AI-native 版本架构图

## 注意事项

- 不是所有东西都需要 AI-native。文件管理、设置页面、导航用代码完全正确。
- 重点审查核心功能，不是边缘功能。
- 不确定 AI 还是代码？问：需要理解上下文吗→AI；需要完全一致→代码；代价大→AI+人在回路。
- **软规则有风险。** 注入、不可复现、模型漂移。确定性场景用硬规则。
- **领域知识优先。** 不熟悉的领域先确认监管要求。
